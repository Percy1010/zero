# 模型上下文协议(MCP)

MCP 是 `Model Context Protocol` 的缩写，中文通常叫“模型上下文协议”。

## MCP 是什么

你可以先把 MCP 理解成：**一种让 AI 系统更标准化地连接外部工具和数据源的通用接口规则**。

## 先用一个类比讲清楚

Anthropic 官方给过一个很适合初学者的比喻：MCP 很像 AI 应用世界里的 `USB-C`。

这个比喻为什么好用？因为你不希望每接一个新设备都重新发明一种接口。你更希望有一套大家都能认的连接标准。

## 为什么需要 MCP

在没有统一协议时，常见问题是：

1. 每接一个新工具都要单独写一套适配逻辑
2. 工具之间的数据格式和调用方式不一致
3. 不同模型、不同 AI 客户端很难复用同一套外部能力

## MCP 的原理

你可以把 MCP 想成三层关系：

1. AI 应用或客户端
2. 协议层
3. 外部工具或数据源

其中协议层的作用是：

- 规定双方怎么描述能力
- 规定怎么交换上下文信息
- 规定怎么请求工具或资源

## MCP 的架构

从实际使用角度，可以先这样理解：

- MCP Host / Client：提出请求的一侧，通常是 AI 应用、Agent 工具或客户端
- MCP Server：把某个工具、资源或系统暴露成标准化接口的一侧

一个常见过程是：

1. AI 客户端连接某个 MCP Server
2. Server 告诉客户端“我能提供哪些工具或资源”
3. 客户端在需要时通过协议发起请求
4. 返回的结果再进入模型上下文，参与后续回答

## MCP 和 Function Calling 有什么关系

- Function Calling 更像“模型请求调用某种外部能力”
- MCP 更像“这些外部能力如何以统一方式被接进来”

所以它们不是互相替代，而是经常配合出现。

## MCP 为什么现在越来越重要

因为 AI 工具已经不再只是聊天窗口，而是越来越像“能接很多系统的工作台”。当 AI 要连接文件系统、文档平台、数据库、设计工具和内部业务系统时，标准化协议就会越来越重要。

## 常见误区

### 误区一：MCP 是某一家公司的私有功能

不是。它更像协议层概念，不应只把它理解成某个单一产品按钮。

### 误区二：有了 MCP，模型就天然更聪明

也不是。MCP 提升的是“接能力的方式”，不是直接改变模型本身的推理质量。

## 你现在只需要记住什么

- MCP 是让 AI 更容易接外部能力的协议
- 它的价值在“标准化连接”，不是“替代模型”
- 先理解 Function Calling，再理解 MCP，会容易很多
