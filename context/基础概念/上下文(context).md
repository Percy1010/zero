# 上下文(Context)

## 一、是什么？

### 一句话定义

`Context` 就是你给 AI 提供的所有背景信息，包括你的问题、历史对话、参考资料、工具结果等一切「信息输入」。  
`Context Window` 则是模型一次最多能看到多少上下文的容量上限。

## 二、工作原理

可以把上下文窗口理解成一张书桌:

- 书桌就是 `Context Window`
- 摆在书桌上的资料就是 `Context`
- 书桌越大，能同时摊开的资料越多
- 书桌满了，就必须删掉一部分、压缩一部分，或者换更大的书桌

所以当一段对话、一个文档、或者一个代码库超出了模型的上下文窗口时，系统就必须截断、摘要、或者拆分，否则模型无法完整处理。

## 三、上下文窗口里到底装了什么？

每次你和模型对话，窗口里装的内容通常比你以为的多得多:

```text
① System Prompt（系统提示词）
② 历史对话
③ 检索到的参考资料（RAG / Search）
④ 工具返回的数据
⑤ 你这一轮的新问题
⑥ AI 当前准备生成的回答
```

重点是: 这些内容都在挤同一个有限窗口。

## 四、不同模型的上下文窗口

| 模型 | 上下文窗口大小 | 大约相当于 |
| --- | --- | --- |
| GPT-3（2022） | ~4K tokens | 约 3,000 字 |
| GPT-3.5 | 16K tokens | 约 12,000 字 |
| GPT-4 Turbo | 128K tokens | 约 96,000 字 |
| Claude 3.5 | 200K tokens | 约 150,000 字 |
| Gemini 1.5 Pro | 1M+ tokens | 约 750,000 字 |

## 五、关键认知: 不是越大就越好

更大的上下文窗口通常意味着:

- 更少的截断
- 更长的对话
- 更强的长文档处理能力

但代价也很明确:

- 成本更高
- 推理更重
- 噪音信息更多时，模型反而更容易「走神」

很多研究都观察到，模型对长上下文中间部分的信息利用并不稳定，往往对开头和结尾更敏感。

## 六、通俗比喻

| 比喻 | 对应解释 |
| --- | --- |
| 书桌 | 决定能同时摊开多少资料 |
| 工作记忆 | 类似人脑一次能抓住多少信息 |
| RAM 内存 | 当前运行中可被系统直接调用的内容 |
| 聚光灯 | 被照到的信息更容易被模型注意到 |

## 七、业务场景应用

| 场景 | 如何利用上下文窗口 |
| --- | --- |
| 客服对话 | 让模型记住客户前面说过的问题和约束 |
| 文档分析 | 直接分析整份合同、报告、方案 |
| 长篇写作 | 保持前后一致的人设、设定和风格 |
| 代码审查 | 一次看到更完整的代码上下文 |

## 八、你现在只需要记住什么

- `Context` 是模型当前真正能看到的信息
- `Context Window` 是它一次能看到多少信息
- 很多回答质量问题，本质上不是模型笨，而是上下文设计得不好
